## Landmines â€” do NOT re-break these
- **Prisma migration drift is reconciled.** A reconciling migration (`20260705095000_batch3_missing_tables_columns`) creates all missing tables (`ApiKey`, `Retest`, `OnboardingState`) and adds all missing columns/indexes/constraints. CI now runs `prisma migrate diff --exit-code` to catch future drift. **Verify on a fresh DB before prod deploy:** `prisma migrate reset && prisma migrate deploy` should reproduce `schema.prisma` exactly.
- **Prisma model sets must match schema columns.** `packages/db/src/scoping.ts` `SOFT_DELETE_MODELS` (19) may contain *only* models with a `deletedAt` column; `WORKSPACE_SCOPED_MODELS` (18) only models with a `workspaceId` column. Adding a model without the column makes the client extension inject a non-existent field â†’ **Prisma throws on every read**. (Real bug once: soft-delete set included `WorkspaceMember`/`CredentialSet`/`AuditLog`/`Retest`, crashing `getWorkspaceMembership`.)
- **Workspace scoping is auto-active.** `requireWorkspaceAccess` calls `setWorkspaceContext(workspaceId)` (AsyncLocalStorage). `WorkspaceMember` (cross-workspace switcher) and `OnboardingState` (per-user) are deliberately excluded from auto-scoping â€” do not add them. Never reintroduce a module-level global for workspace context (cross-request tenant leak).
- **Postgres RLS is live.** RLS enabled on all 18 workspace-scoped tables with permissive (no context = allow all, backward compat) + strict (context set = only matching `workspaceId`) policies. Use `withWorkspaceRLS(workspaceId, fn)` from `@lyrashield/db` to wrap queries that need DB-level isolation. The helper uses `SET LOCAL` inside a transaction (connection-safe with Prisma pooling). CI validates the migration on Postgres 16.
- **Workflow files need special GitHub permission.** The Hyperagent GitHub app lacks `Workflows: write`, so agents using it get 403 writing `.github/workflows/*`. Route CI/workflow edits to a user-credentialed agent (e.g. Codex) or the repo owner.
- **Rename is partial.** Repo is `ecryptoguru/lyrasec-ai`; product is migrating LyraShield â†’ LyraSec. **Do NOT rename** `@lyrashield/*` package scopes or `LYRASHIELD_*` env vars without founder sign-off (trademark clearance open). Keep everything domain-agnostic (`NEXT_PUBLIC_APP_URL`); public domain is undecided.
- **A4 follow-up:** store the numeric GitHub `installationId` on `Target` and match on it (webhook currently matches by `repoFullName` owner-prefix).
- **Schedule.target FK migration exists.** Migration `20260706010000_schedule_target_fk` adds the `Schedule_targetId_fkey` foreign key. Do not drop or recreate it. If you add a new migration that touches the `Schedule` table, ensure it doesn't conflict with this constraint.
- **Worker notifications have no `userId`.** Notifications created by the worker (`notifyScanCompleted`, `notifyScanFailed`, `notifyCriticalFinding`) are workspace-level â€” they do not set `userId`. The notifications list API must not filter by `userId` by default, or worker-created notifications will be invisible. Use the optional `userId` query param only when user-specific filtering is needed.
- **`createAndSendNotification` is the shared notification helper.** Located in `packages/db/src/notification-service.ts`. It accepts a `sendFn` callback to avoid circular dependencies between `@lyrashield/db` and `@lyrashield/integrations`. Both the worker (`apps/worker/src/notifications.ts`) and the create-pr API route use it. Do not duplicate the create+send loop â€” use this helper.
